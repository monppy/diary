1. プロダクトの概要と目的

  目的: 振り返り習慣を構築し、同じ失敗を繰り返さないための個人用日記アプリ。

  コアバリュー:
  - 日々の出来事・気づきを手軽に記録する
  - 過去の日記をタグ・キーワード・日付範囲で検索して振り返れる
  - 自由記述とテンプレート（今日やったこと / 思ったこと / 明日に活かすこと）の2モードを提供する

  ユーザー: 個人1名のみ。マルチテナントではない。

  4. データベーススキーマ構造

  ER 図（概念）

  auth.users (Supabase管理)
      │ userId (UUID)
      │
      ├──< Entry (日記)
      │      id        TEXT PK
      │      userId    TEXT  ← auth.users.id を文字列で保持（外部キー制約なし）
      │      entryDate DATE
      │      body      TEXT
      │      mode      EntryMode ENUM ('FREE' | 'TEMPLATE')
      │      templateKey TEXT? （将来の複数テンプレ識別用）
      │      createdAt TIMESTAMP
      │      updatedAt TIMESTAMP
      │
      ├──< Tag (タグ)
      │      id        TEXT PK
      │      userId    TEXT
      │      name      TEXT  ← # を除いた名前
      │      createdAt TIMESTAMP
      │      UNIQUE (userId, name)
      │
      └── EntryTag (中間テーブル)
             entryId  TEXT FK → Entry.id CASCADE
             tagId    TEXT FK → Tag.id  CASCADE
             PK (entryId, tagId)

  インデックス戦略

  ┌───────────────────────────────┬─────────────────────────────────────────┐
  │         インデックス          │                  目的                   │
  ├───────────────────────────────┼─────────────────────────────────────────┤

  ├───────────────────────────────┼─────────────────────────────────────────┤
  │ Entry(userId, createdAt DESC) │ 作成日時での並び替え                    │
  ├───────────────────────────────┼─────────────────────────────────────────┤
  │ Tag(userId)                   │ タグ一覧取得・オートコンプリート        │
  ├───────────────────────────────┼─────────────────────────────────────────┤
  │ Tag(userId, name) UNIQUE      │ タグ重複防止・upsert                    │
  ├───────────────────────────────┼─────────────────────────────────────────┤
  │ EntryTag(tagId)               │ タグからエントリを逆引き                │
  ├───────────────────────────────┼─────────────────────────────────────────┤
  │ Entry.body GIN trigram        │ 日本語本文の部分一致全文検索（pg_trgm） │
  ├───────────────────────────────┼─────────────────────────────────────────┤
  │ Tag.name GIN trigram          │ タグ名の部分一致検索                    │
  └───────────────────────────────┴─────────────────────────────────────────┘

  テンプレート本文のフォーマット規約

  テンプレートモードの本文はデータベース上は単一の TEXT として保存される。フォーマット:

  ## 今日やったこと
  {テキスト}

  ## 思ったこと
  {テキスト}

  ## 明日に活かすこと
  {テキスト}

  UI 側でパース/直列化を担い、DB はモード判定フラグ（mode = 'TEMPLATE'）のみ保持する。

  5. 認証・セキュリティ機構

  認証フロー全体図

  ユーザー
    │
    │ 1. メールアドレスを入力
    ▼
  [/login] Server Action: loginAction()
    │ supabase.auth.signInWithOtp({ email, emailRedirectTo: '/auth/callback' })
    │
    ▼
  Supabase Auth サーバー
    │ 2. マジックリンクをメール送信
    ▼
  ユーザーのメール受信トレイ
    │ 3. リンクをクリック
    ▼
  [/auth/callback?code=xxx] Route Handler
    │ supabase.auth.exchangeCodeForSession(code)
    │ → セッション Cookie を発行
    ▼
  [/entries] ログイン完了

  セッション情報の格納場所

  ┌──────────────────────────┬────────────────────────────────────────────────┬───────────────┐
  │           情報           │                    格納場所                    │    管理者     │
  ├──────────────────────────┼────────────────────────────────────────────────┼───────────────┤
  │ ユーザーアカウント       │ Supabase auth.users テーブル（非公開スキーマ） │ Supabase      │
  ├──────────────────────────┼────────────────────────────────────────────────┼───────────────┤
  │ JWT アクセストークン     │ HTTP Only Cookie (sb-*-auth-token)             │ @supabase/ssr │
  ├──────────────────────────┼────────────────────────────────────────────────┼───────────────┤
  │ リフレッシュトークン     │ HTTP Only Cookie                               │ @supabase/ssr │
  ├──────────────────────────┼────────────────────────────────────────────────┼───────────────┤
  │ アプリ固有ユーザーデータ │ なし（auth.users.id を userId として参照）     │ アプリ        │
  └──────────────────────────┴────────────────────────────────────────────────┴───────────────┘

  アプリは auth.users を直接管理しない。Prisma スキーマに User テーブルを持たず、Entry.userId と Tag.userId に       
  auth.users.id（UUID 文字列）を保存するのみ。

  サーバー側の認証確認（多層防御）

  Layer 1: ミドルウェア (src/middleware.ts)
  全リクエスト
    └─ supabase.auth.getUser() ← Supabase Auth サーバーに問い合わせ（JWT検証）
         ├─ 未認証 & 非公開パス → /login にリダイレクト
         └─ 認証済み & /login → /entries にリダイレクト
  getSession() ではなく getUser() を使用している。getSession() は Cookie の値をそのまま返すため JWT
  の真正性を検証しない（SSRF やクッキー偽造に脆弱）。

  Layer 2: Server Action (src/actions/entries.ts)
  async function getAuthenticatedUserId(): Promise<string> {
    const supabase = await createClient();
    const { data, error } = await supabase.auth.getUser(); // 再度サーバー検証
    if (error || !data.user) throw new Error("Unauthorized");
    return data.user.id;
  }
  各 Server Action の先頭でユーザーIDを取得。

  Layer 3: データアクセス層 (src/lib/db/entries.ts)
  // getEntry / deleteEntry / updateEntry で userId を照合
  if (!existing || existing.userId !== userId) return null;
  DB から取得した行の userId と認証済みユーザーの ID を比較。他ユーザーのデータは絶対に返さない。

  Supabase SSR のセッションリフレッシュ

  @supabase/ssr は JWT の有効期限切れを検出すると、リフレッシュトークンを使って新しいアクセストークンを取得し、setAll
   コールバック経由でレスポンス Cookie に書き戻す。これがミドルウェアで必ず実施される理由であり、Cookie
  の書き込み権限をミドルウェアだけに任せている。

  入力バリデーション (Zod)

  全 Server Action の入力は必ず Zod スキーマで検証してから DB に渡す:
  - フォーマット違反（日付形式・文字数超過等）→ 即時 fieldErrors 返却
  - タグ名は # 除去・trim・空文字チェックを自動適用

  6. 開発中に直面した主な技術的課題と解決策

  課題 1: Prisma 7 の破壊的変更 — url/directUrl が schema.prisma から廃止

  状況: pnpm prisma format を実行すると P1012: url is no longer supported in schema files エラー。

  原因: Prisma 7 は接続 URL の設定先を prisma.config.ts に移管した。schema.prisma の datasource ブロックに url =     
  env("DATABASE_URL") を書く従来の方法が使えなくなった。

  解決策:
  // prisma.config.ts
  export default defineConfig({
    datasource: {
      url: process.env.DIRECT_URL ?? process.env.DATABASE_URL, // マイグレーション用
    },
  });
  また Prisma 7 では runtime クエリに @prisma/adapter-pg を PrismaClient コンストラクタに渡す新方式が必須になった。  

  ---
  課題 2: Supabase の direct 接続 URL からの接続不可

  状況: DATABASE_URL に db.<ref>.supabase.co:5432 を設定したが P1001: Can't reach database server エラー。

  原因: Supabase の旧 direct 接続（db.xxx.supabase.co）は IPv6 のみ対応のため、IPv4 しか持たない環境（Windows +      
  一般的な ISP）からは到達不能。

  解決策: Supabase Supavisor（pooler）の URL に切り替え:
  - DATABASE_URL: aws-1-ap-southeast-2.pooler.supabase.com:6543（Transaction Pooler・runtime 用）
  - DIRECT_URL: aws-1-ap-southeast-2.pooler.supabase.com:5432（Session Pooler・migrate 用）

  ---
  課題 3: prisma migrate dev が advisory lock で詰まる

  状況: prisma migrate dev プロセスをキャンセルした後、次の migrate deploy が SELECT pg_advisory_lock(72707369)      
  でタイムアウト。

  原因: Supavisor がセッションをプールしているため、強制終了したプロセスが持っていた advisory lock
  がセッション終了まで解放されなかった。

  解決策: 3ステップで復旧:
  1. prisma db execute でロックを保持しているバックエンドを pg_terminate_backend() で強制終了
  2. prisma db execute --file migration.sql で SQL を直接適用
  3. prisma migrate resolve --applied <名前> で _prisma_migrations テーブルに適用済みとして記録

  以降のマイグレーション追加は migrate dev --create-only → SQL 編集 → db execute → migrate resolve --applied の手順が
   Supabase 環境での安定パターン。

  ---
  課題 4: shadcn/ui base-nova スタイルが asChild prop 非対応

  状況: <Button asChild><Link href="..."> の型エラー。asChild が ButtonProps に存在しない。

  原因: 最新の shadcn/ui base-nova スタイルは Radix UI の Slot ではなく @base-ui/react/button
  をベースとするため、asChild パターンが使えない。

  解決策: ボタンスタイルのリンクには buttonVariants() ユーティリティと <Link> を直接組み合わせる:
  import { buttonVariants } from "@/components/ui/button";
  <Link href="/new" className={cn(buttonVariants({ size: "sm" }))}>
    + 新規
  </Link>

  ---
  課題 5: 日本語全文検索に Postgres 標準 tsvector が使えない

  状況: 日本語はスペース区切りがなく、to_tsvector('japanese', body) は適切にトークナイズできない。

  解決策: pg_trgm 拡張の GIN trigram インデックスを使用:
  CREATE EXTENSION IF NOT EXISTS pg_trgm;
  CREATE INDEX "Entry_body_trgm_idx" ON "Entry" USING GIN (body gin_trgm_ops);
  Prisma の body: { contains: keyword, mode: "insensitive" } が生成する ILIKE '%keyword%' クエリに対して、この GIN   
  インデックスが効く。将来的に性能不足になれば pg_bigm や pgroonga への移行を検討する。

  ---
  課題 6: Next.js 16 で params/searchParams が非同期になった

  状況: ページコンポーネントで params.id にアクセスすると TypeScript エラー。

  原因: Next.js 15 以降、params と searchParams は Promise<...> になった。

  解決策: 全ページで await params / await searchParams:
  export default async function Page({ params }: { params: Promise<{ id: string }> }) {
    const { id } = await params;
  }

  ---
  課題 7: Zod v4 で z.nativeEnum() が廃止

  状況: Zod v4.3.6 がインストールされ、z.nativeEnum(EntryMode) に deprecation 警告。

  原因: Zod v4 は z.enum() が const オブジェクト（EnumLike）を受け付けるようになり、nativeEnum を吸収した。

  解決策: z.enum(EntryMode) に統一（EntryMode は Prisma 生成の const オブジェクト）。
